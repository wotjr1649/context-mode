# v1.0.3 — upgrade/update subsystem charter repoint (design)

**Status:** design, approved in brainstorm (2026-07-12). Feeds writing-plans. Line
refs drift — locate by symbol.

**Context.** The 1.0.2 pre-commit Codex review found the fork's whole
upgrade/update subsystem still pointed at UPSTREAM (`mksglu/context-mode`) — a D9
charter blocker: the documented `/ctx-upgrade` could overwrite the hard fork with
upstream 1.0.169 and flood tags. 1.0.2 shipped only the MINIMAL neutralize
(repointed the two `git clone` URLs `mksglu`→`wotjr1649`, so `ctx upgrade` clones
the fork, not upstream). This spec completes the charter fix.

**Goal.** Remove every remaining upstream-pointing path from the update/upgrade
subsystem, and add a build gate that keeps upstream refs out of runtime source.

**Non-goals.**
- No change to the marketplace `/plugin update` flow (the proven I1 deploy path).
- No replacement update-notification mechanism (decided: drop it — see C1).
- No touch to the clone URLs already repointed in 1.0.2.
- No broad bare-`#NNN` comment cleanup (declined in 1.0.2 — not charter funnels).

## C1 — Remove the version-check / update-notification (upstream npm)

The check GETs `https://registry.npmjs.org/context-mode/latest` = **upstream's**
published version (the fork is `private`, unpublished), producing a misleading
"→ v1.0.169" indication in **two** surfaces, both of which must go:
1. the per-tool-call **nag banner** (server.ts tool-response prepend), and
2. the **ctx_stats report** ("update available" line, fed by `_latestVersion`).

Decision (brainstorm): **remove it entirely.** A private single-maintainer fork
has no npm "latest"; users update via `/plugin update` (marketplace = I1). No
fork-GitHub replacement (YAGNI).

Remove (all in cli.ts / server.ts unless noted):
- `fetchLatestVersion()` in **both** files, and the server.ts startup calls
  (`fetchLatestVersion().then(v => _latestVersion = v)`).
- Nag banner + its nag-only helpers: `getUpgradeHint`, `isOutdated`,
  `shouldShowVersionWarning`, the **server-local** `semverNewer`, and the burst
  state (`_warningBurstCount`, `_lastBurstStart`, `VERSION_BURST_SIZE`,
  `VERSION_SILENT_MS`).
- `_latestVersion` state: pass `null`/omit to the three `formatReport(…,
  _latestVersion, …)` calls so the stats "update available" line never renders.
  analytics.ts keeps its own `latestVersion` param + its own `semverNewer` guard
  (they simply never fire with `null`) — a harmless dead param; pruning it is an
  optional later cleanup, out of scope here.
- cli.ts doctor/status: drop the `latestVersion` comparison branches; report
  local + adapter-installed version only.

Keep: the `upgrade()` command's "Update available: v<local> → v<new>" message —
it compares against the FORK clone (post-1.0.2 repoint), not upstream npm.

## C2 — Marketplace sync hardening (cli.ts "Step 0")

`git fetch --tags origin` on the marketplace clone floods tags (charter-forbidden)
and `git reset --hard origin/HEAD` trusts whatever `origin` is.
- Drop `--tags` (fetch the branch ref only).
- BEFORE fetch/reset, validate the marketplace clone's `origin` URL resolves to
  the fork (`wotjr1649/context-mode`); if not, **skip the sync** with a log line
  (fail-safe — never hard-reset a clone pointed elsewhere).
- Extract the origin-is-fork check as a pure function + unit test.

## C3 — Static charter gate (new `scripts/assert-no-upstream-mksglu.mjs`)

A build-time assert (wired into `npm run build` alongside the existing `assert-*`)
that FAILS if `mksglu/context-mode` (clone URL, issue link, or bare repo ref) or
`registry.npmjs.org/context-mode` appears in runtime source: `src/`, `hooks/`,
`scripts/`. Excludes `*.bundle.mjs` (generated), `docs/`, `tests/fixtures/` —
mirroring `assert-no-removed-platforms`'s exclusion pattern. New script (separate
concern from the removed-platform ids). Locks in the 1.0.2 clone repoint + this
cleanup, and subsumes the deferred "charter guard for issue-links."

## Error handling / fail-safe

All of C1 is surface REMOVAL — the version-check is informational only; nothing
branches on it functionally, so removal cannot change behavior. C2 fails safe
(skip sync on non-fork origin). C3 is build-time only. Net: strictly less runtime
surface + a permanent guard.

## Testing

- **C2**: pure origin-is-fork predicate unit test (fork URL variants — https/ssh,
  with/without `.git` — pass; upstream + arbitrary remotes fail).
- **C3**: the gate self-verifies (like assert-no-removed-platforms); a mutation
  check during implementation confirms a planted `mksglu/context-mode` string
  trips the build.
- **C1**: update affected doctor/status + stats/nag tests; confirm the capped
  suite stays ⊆ the pre-existing baseline (no new regression).

## Deploy (1.0.3)

Same proven cycle: hand-bump package.json 1.0.2→1.0.3 → `npm run version-sync` →
`npm run build` (now including the C3 gate) → gate (typecheck + capped suite) →
commit → `git push origin main` + `git push origin refs/tags/v1.0.3` (single ref,
never `--tags`) → `claude plugin marketplace update context-mode-js` → `claude
plugin update context-mode@context-mode-js` → `node scripts/verify-deploy.mjs
1.0.3` PASS → reload. **Codex cross-model review** at plan-ready (adversarial —
focus: does the removal miss an upstream surface? does C2 origin-validation have a
bypass?) + pre-commit diff.

## Acceptance criteria

- No runtime source (`src/`/`hooks/`/`scripts/`) contains
  `registry.npmjs.org/context-mode` or `mksglu/context-mode` — C3 gate green,
  residue 0.
- After deploy + reload, no "update available → v1.0.169" appears in any ctx tool
  output nor in ctx_stats.
- Marketplace sync no longer passes `--tags` and skips on a non-fork origin.
- Capped suite ⊆ pre-existing baseline; `verify-deploy 1.0.3` PASS.
