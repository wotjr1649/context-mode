# v1.0.2 hardening — post-1.0.1 remediation design

**Status:** design only. 1.0.1 shipped and I1 is CONFIRMED (version bump → push →
`plugin update` reinstalled the deployed tree at 1.0.1, verified byte-identical to
`351587a`). **Every residual below is fail-safe** — nothing in 1.0.1 is broken.
This is the prioritized backlog for a *deliberate* 1.0.2 pass, not urgent work.

> Sources: the pre-deploy adversarial panel (Claude A/D + 3 cross-model Codex
> passes) and `docs/superpowers/phase-4-plus-candidates.md`. Line refs drift —
> locate by symbol.

## How a 1.0.2 would ship

Same flow that worked for 1.0.1 (now proven end to end):
hand-edit `package.json` → `npm run version-sync` → `npm run build` (bundles) →
gate (typecheck + capped suite ⊆ 6 pre-existing) → commit → `git push origin main`
+ `git push origin refs/tags/v1.0.2` (single refspec) → `claude plugin marketplace
update context-mode-js` → `claude plugin update context-mode@context-mode-js` →
`node scripts/verify-deploy.mjs 1.0.2` (PASS) → restart to load. Test-only items
(P1.2, P2.3) need **no** re-deploy.

## P1 — highest value

### P1.1 — deps-heal internal timeout vs host hook budget  *(needs re-deploy)*
`hooks/deps-heal.mjs` install `timeout: 180000` ms, but the SessionStart host
budget is ~60 s and neither `hooks/hooks.json` nor `.codex-plugin/hooks.json`
sets a per-hook override. Flagged by D (rounds 1 & 2) and Codex. Today it is
fail-safe (a slow heal is host-killed → partial rm'd → retried next session), but
capping the internal timeout below the host budget (~45 s) makes the internal
timeout *throw → caught* instead of an abrupt host kill, so the log line is
written and the state is predictable. ~1-line change. The 3 target packages
(turndown, turndown-plugin-gfm, @mixmark-io/domino) are small pure-JS, so 45 s is
ample.

### P1.2 — verify-deploy CLI integration tests  *(test-only, no re-deploy)*
`verifyDeploy` + `isVersionDirUnderCache` are unit-tested (23 cases incl. the
containment/traversal/half-install/multi-entry guards), but the CLI wrapper —
exit codes 0/1/2, FATAL on missing/malformed `installed_plugins.json`,
`CLAUDE_CONFIG_DIR` resolution, and the `realpathSync` reader — is only covered by
the manual pre-flight. verify-deploy is now the **standing release gate** for
every future 1.0.x, so it earns spawn-the-script tests: a temp `CLAUDE_CONFIG_DIR`
with a crafted `installed_plugins.json` + a fake cache tree, asserting exit
0/1/2 and the PASS/FAIL/FATAL text. Add a junction-based case for the realpath
symlink guard on win32.

## P2 — medium value

### P2.1 — mksglu upstream links in scripts  *(needs re-deploy; charter)*
`scripts/postinstall.mjs` and `scripts/heal-better-sqlite3.mjs` reference
`github.com/mksglu/context-mode/issues/*` in comments/diagnostics. Not invoked by
any runbook command and not an infra call, but they funnel issues upstream against
the D9 charter ("no links funnelling issues upstream"). Repoint at
`wotjr1649/context-mode` or drop. Trivial.

### P2.2 — orphan-reaper isReapable substring + TOCTOU  *(needs re-deploy)*
`isReapable` is a full-command-line substring test: a dead-parent, >60 s,
cache-root-referencing process (an editor/grep/tail with the cache path in its
args) would match, and the 20 s-old snapshot's PIDs are killed without
re-validating identity (Windows recycles PIDs). Ships **inert** (DRY_RUN unless
`CONTEXT_MODE_REAPER_ARMED=1`), so this only bites if a user ever arms it. Harden
by matching the *executable* (node/pwsh) under the cache root rather than any
substring, and re-confirming the target's CommandLine/CreationDate immediately
before `process.kill`.

### P2.3 — tests/ type debt (157 errors)  *(test-only, no re-deploy)*
`npm run typecheck:tests` surfaces ~95 TS7016 (missing `.mjs` declarations —
convention is `// @ts-expect-error — JS module, no TS declarations`), ~12 TS2578
(unused directives), and a handful of genuine fixture-shape mismatches
(`tests/analytics/format-report.test.ts`, `tests/benchmark.ts`, missing `.js`
import extensions in `tests/adapters/*`). Burn down the genuine mismatches first;
blanket-suppress or `checkJs:false` the .mjs-declaration noise.

## P3 — opt-in / own plan

- **npmCliPath robust discovery** *(re-deploy)* — 1.0.1 reverted to 2 execPath
  candidates + fail-closed null (a non-standard node layout no-ops). A robust
  discovery must parse PATH **without** running a locator (Windows `where`
  searches CWD first — the reverted RCE) and verify `realpath` lands under a
  trusted Node / version-manager root. Only worth it if non-standard layouts (nvm/
  fnm/volta/scoop) become a real target.
- **deps-heal transactional heal** *(re-deploy)* — the benign rm-then-fail (a
  transient install failure loses an already-broken partial, self-heals next
  session) could be made transactional, but round-2's rename-aside proved this
  needs a per-run backup name + an install-root lock to avoid a concurrent-
  SessionStart race — non-trivial; only if the fail-safe loss ever matters.
- **init-time hard-fail (net-4 completion)** *(re-deploy)* — the removed-client
  `UnsupportedClientError` fires at request boundaries, not MCP init
  (`getClientVersion()` is undefined at connect). Await initialization in `main()`
  so a removed client hard-fails at boot; add a source-contract test pinning the
  boot-path rethrow. Behavior change → its own small plan.
- **f*.tmp root cause** *(test-only)* — a test writes `f0/f1/f2.tmp` cwd-relative
  each run (gitignored, harmless). Redirect it to a temp dir.
- **SAFE_RANGE `|`** — now moot: the `node <npm-cli.js>` invocation removed
  cmd.exe from the path, so `| > < ^` are inert argv. No action; recorded so it is
  not re-opened.
